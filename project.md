## §1 核心原则（按需触发）

| 原则 | 一句话 | 触发条件 |
|------|--------|----------|
| P1 三层分离 | 数据层→生成层→输出层，各层独立迭代 | 产出超AI单次上限，或含确定性计算/渲染 |
| P2 边际成本递减 | 重复劳动沉淀为脚本/模板/清单 | 发现在重复写prompt/手动调格式 |
| P3 质量可检查化 | "做得好"转为可勾选清单 | 主观质量评价出现时 |
| P4 先验证再规划 | 先获真实数据结构再写方案 | 依赖外部系统 |
| P5 规则从简 | 先最简版本，遇问题再加条件（P5管实现路径，铁律-1管产出质量，不矛盾） | 设计业务逻辑规则时 |
| P6 一次一事 | 一个对话一件事；超20轮开新对话 | 任何对话 |
| P7 先理解再动手 | 搞清架构全貌再操作 | 部署新平台 |
| P8 五个元问题 | 跑在哪/谁在用/数据在哪/怎么触发/输出给谁 | 新项目启动 |
| P9 从输出倒推输入 | 先定义最终看到什么，反向追踪 | 设计数据处理/报表 |
| P10 修复最小改动 | 超5处改动先想是不是结构问题 | 修改已有文档/代码 |
| P11 路径受阻即切换 | 走不通是切换信号不是停止信号 | 遇到阻碍 |

## §2 通用开发六阶段

### 项目类型差异速查

| | AI内容生成 | 工具/爬虫 | AI Skill | Agent部署 | 数据系统 | 视频/图像 |
|---|---|---|---|---|---|---|
| 核心挑战 | 输出质量控制 | 外部系统对接 | 指令穷尽性 | 资源管理+组件协调 | 数据流完整性 | 算法设计完备性 |
| 最大风险 | 截断/格式不稳定 | 接口猜错/反爬 | section间逻辑不一致 | context overflow/成本失控 | 外部依赖变更/数据膨胀 | 边界case爆炸/阈值脆弱 |
| 前期重点 | Prompt+输出格式+视觉方案（如有视觉产出→触发visual-design.md） | 技术探查+接口验证 | 真实数据+边界穷举 | 理解平台架构+资源预算 | 五元问题+领域认知 | 算法穷举设计+真实样本验证+视觉标准（触发visual-design.md §4 Phase 1-2） |
| 产出物 | PDF/报告 | 可运行代码 | 指令文档 | 可自主运行的agent | PRD+CLAUDE.md | PRD(含完整算法)+CC按Phase开发 |

**阶段一·需求讨论。** 先"为什么"再"怎么"。确认P8五元问题+产品形态。涉及专业领域先花1-2轮建领域认知。

**阶段二·技术探查。** 涉及外部系统不可跳过。记录结果标注[已验证]。涉及外部数据入库→出口门禁含数据质量诊断（空记录率/完整性/格式一致性量化报告），不达标不进下一阶段。

**阶段三·需求文档。** 技术断言标来源，关键逻辑附10+测试用例。PRD核心：意图+规则双层表达、公式给完整计算示例、边界穷举、DRY、外部依赖写降级方案、领域种子数据由用户提供完整内容（不得写"参照格式补全"——CC无法凭空生成领域知识）、验收标准=可执行命令+可观测结果含期望数字。交付前过检查清单（§3）。

**阶段四·多轮审查。** 每轮换视角（功能→业务→异常→技术→端到端→实现者→模拟运行365天）。每2-3轮精简一次。上限3轮，超过改结构不打补丁。

**阶段五·开发。** CC先验证所有[待验证]项。一个对话只做一件事。

**阶段六·复盘。** 对比计划与实际，经验写入changelog。

### 各类型要点

**AI内容生成：** 跑通链路(1-2天容忍粗糙)→逐项打磨(严格按序：分析→内容→排版→文风，每轮只改一类)→跑通+3轮后必须拿给真实用户。分析层和输出层严格分离。从零生成而非增量修改。

**工具/爬虫：** 写代码前先手动操作一遍。渐进式：跑通一次→确认数据结构→加翻页去重→最后做前端和错误处理。防御性：每步边界检查、重试超时、断点续做。

**Agent部署：** 先理解平台完整架构(P7)——文件加载/进程模型/认证/配置schema/context管理。Context预算是硬约束。成本先算再跑。进程唯一性。

**数据处理/监控：** 先确认P8尤其"跑在哪里"。先建领域认知。schema先于一切。外部API三层隔离。写入默认幂等。从输出倒推输入(P9)。

**视频/图像处理：** 算法设计必须在写代码前完成到可直接翻译为代码的程度。阈值多区间设计。必须做错误传播分析。多层独立防御。逐Phase开发验证。

## §3 检查清单

### 文档交付检查清单

所有技术断言标来源 · 关键逻辑附10+测试用例 · API定义完整 · 错误处理覆盖已知异常 · 无歧义表述 · 从"CC会卡在哪"角度审查过 · 公式有完整数值示例 · 区间无缝隙且NULL有处理 · DRY · 外部依赖有降级方案 · 种子数据完整（无伪完成） · 验收标准有期望数字

### 部署检查清单

编码一致 · 多语言文本正确 · 路径无特殊字符 · 运行环境就绪 · 端口未占用 · 防火墙不阻拦

### Agent部署检查清单

平台架构已理解 · 无其他进程监听同通道 · workspace常驻<context 5% · 大文件按需加载 · API成本已估算 · 配置已备份 · 重启命令已记录

### 视频/图像处理检查清单

核心算法已设计到可直接翻译为代码的程度 · 阈值判定采用多区间设计 · 错误传播路径已分析 · 每种错误类型有检测和防御机制 · 地基模块有独立自检机制 · 反向检查已设计（最终结果回溯到原始输入） · 测试样本覆盖多种难度 · 可替换模块设计为接口+实现分离 · 每Phase有可视化验证标准 · Phase总结机制已建立

## §4 模板

### 排版规范模板

```
【布局】横版A4，每页一个核心观点，留白充分
【对齐】卡片上下沿对齐，页边距全文一致，内容量均匀分布
【字号】正文≥12.5px，标题≥16px
【配色】克制专业色系。主色+辅助色+中性灰，≤4色
【优先级】图示 > 表格 > 要点卡片 > 段落文字。能可视化的不用文字
【页数】无上限，每页有信息增量
```

### PRD结构模板

```
§0 开发者必读（领域知识30秒版+意图+术语+常见误解）
§1 系统概述（一句话+部署环境+用户故事）
§2 文件结构+执行流程DAG
§3 数据库Schema（唯一真相源）
§4 外部数据源（API签名+字段映射+返回示例+降级）
§5-§N 各模块详细逻辑（输入/输出/异常/伪代码）
§N+1 输出层（每个数字的数据来源追溯）
§N+2 初始化/冷启动
§N+3 容错矩阵
§N+4 配置参数集中定义
§N+5 开发优先级（Phase+可执行验收命令+自检协议）
§N+6 依赖（精确版本）
附录A 常见陷阱（错误→正确对照）
附录B 扩展指南
配套：CLAUDE.md指示CC读PRD按Phase开发
```

### 文风规范模板

```
【指代】首次全称，后续统一"其"
【句式】结构完整，不用省略句
【禁用】破折号解释性插入、非必要引号、翻转句式、填充句式
【语体】专业评论水准。专业但不晦涩，克制但不干涩
【数据】每个定性判断附定量支撑。精度匹配样本量
【立场】不预设读者身份，保持中立
```

### 区间规则约定

连续值左闭右开，整数闭区间，NULL必须定义默认值。有显式标注以标注为准。
