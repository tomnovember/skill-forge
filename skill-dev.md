# skill-dev.md · Skill设计与打磨

---

## §1 前置步骤

### 领域探索

新领域先花1-2轮建立基本认知再做立项决策。让AI输出结构化的领域知识手册，用户审核是否正确。1小时领域认知可节省10小时返工。

### 立项决策

三个条件同时满足才做：频率（年50次以上值得轻量，100次以上值得工程级）、可编码性（专业判断能否结构化为规则和流程）、质量杠杆（Skill能把质量上限推到什么水平）。

ROI速算：(每次节省时间×年使用次数×时间价值) / 构建成本。

### 对标

找真实样本建立质量基准。质量的锚点是"该领域最好的产出物长什么样"，不要用"AI能做到什么水平"当基准。

## §2 工作流程

### 入口判断

"帮我做个Skill"→问"有积累还是从零？"有积累→模式C，从零→模式A。"Skill有问题"→模式B。"把这个固化"→模式C。"看看这个Skill"→用quality.md逐条检验，输出评估报告。

### 模式A：从零设计

**定锚→初稿→测试迭代→精简→定稿。**

定锚钉三件事：文风锚点（一句话语体标准）、产出物边界（输入/输出/不做什么）、触发条件（宁可pushy多触发）。

初稿跟快节奏——出框架、暴露问题。搭框架时做职责分离：领域知识和处理流程归Skill，交互风格和用户偏好归Instruction。

审查以真实数据驱动：真实输入跑完→发现问题→定位规则缺失还是冲突→修→换不同输入再跑→连续两个不同输入无新问题才完成。跑不了真实数据用穷尽性五维度思维实验，标注"未经实测"。

精简每2-3轮做一次：补丁转正面规则、合并重复、删冗余（遮住还能做对的就是冗余）、排查矛盾。审查超3轮仍有新问题→结构问题，改结构不打补丁。总轮次上限5轮，超过重新定锚。二次定锚后仍不收敛→讨论拆分或降标。

路径走不通→路径切换信号，换路继续推进，说明原因即可。

### 模式B：修Skill

**产出是最可靠的诊断入口。** 先区分两种症状：

**产出明确错误**（每次都有同样问题）→规则缺失或冲突。带着产出回溯规则→诊断→输出完整修改版（不是diff）。

**产出不稳定**（时好时坏）→规则模糊地带。拿好产出和坏产出对比，找分叉点，收窄选择空间。

### 模式C：从项目提炼（最常用）

回溯对话提取三类信息（用户修正=规则、边界case=测试用例、明确偏好=硬约束）→区分普遍和特例→真实数据验证→用户确认。

### 模式间切换

不符就切，说一句原因即可。

### 固化时机

同一操作重复3次以上、修正开始收敛、流程能产出满意产出物→主动建议固化。

## §3 架构设计

**单文件vs多文件：** 知识量<50KB且流程≤5步→单文件；否则多文件。

**三种模式：** Round制（传数据）、阶段制（传认知，驱动力是改变核心判断）、知识库制（按需加载）。可混用。

**主控文件≤100行纯路由。** SKILL.md管"做什么、什么顺序、谁决定"，其他文件各管专门职责。任何规则只在一个文件定义，其他只引用。超100行检查：是否有内容混入了路由？有→拆。

**context管理：** 标注每个文件常驻/按需，按频率分库，主控保持轻量。区分Skill中不变的静态内容（铁律、规则、知识库）和随输入变化的动态内容（当前分析对象、用户选择、中间状态），静态部分可以更长，动态部分严格压缩。

### 知识结构化

三种方式：知识卡片（统一格式标注来源时效）、基准数据表（分维度分类型标注校准时间）、视角/模型库（标注核心问题和适用对象）。

拆成结构化条目不存大段文字。知识来源必须是权威一手源。时效管理：标注校准时间+超期强制重验证。

### 流程设计

认知生产系统定位：驱动力是"改变核心判断"不是"步骤做完"。

多步骤流程中标注每步的完成标准（什么证明这步做完了可以往下走）和思考深度（机械执行还是深度思考——防止简单步骤浪费资源、关键步骤思考不足）。涉及不可逆操作的步骤标注human checkpoint。简单流程（≤3步）不需要逐步标注。

非线性回退：任何时刻最弱点在哪回哪。

打磨终止：连续两轮核心判断无实质性改变。

异常中断：发现重要异常立刻追，不等当前环节结束。

强制执行机制："请务必执行"不如"不执行就过不了关"。两种已验证方式——执行记录字段为空不得通过；门禁强制生成型输出不通过不往下走。

## §4 规则设计

**铁律体系：** 铁律-1永远追问极限为标配。铁律零防编造。其余按风险排序≤5条。数量与风险结构匹配。

**溯源测试：** 终检时逐句执行，每条陈述能否指向来源？指不出→删或标注。

**定位边界声明：** 涉及用户专业判断的Skill必须声明"辅助不代替"。

**模块配置：**

| 模块 | 用途 | 配置依据 | 详见 |
|------|------|---------|------|
| A 信息检索 | 搜索策略与路径管理 | 需要外部信息 | `methods.md §1` |
| B 事实校验 | 防幻觉+三层攻击 | 所有Skill（标配） | `expression.md §2` |
| C 可信度分层 | 来源排序与冲突处理 | 多源信息 | `expression.md §3` |
| D 表达质量 | 去AI味+感性标准 | 有自然语言输出 | `expression.md §1` |
| E 终检 | 一致性检查 | 所有Skill（标配） | `quality.md §4` |
| F 经验积累 | 过拟合监控 | 持续迭代 | `methods.md §3` |
| G 降级处理 | 异常与信息不足 | 依赖外部数据 | `expression.md §4` |
| H 质量检查 | 增量/全量检查 | 所有多文件Skill | `quality.md §3` |
| V 视觉设计 | 视觉决策逻辑+审美迭代+逐维度并排 | 有视觉产出 | `visual-design.md §4` |

## §5 定稿检查与质量门禁

### 定稿检查清单

- 产出水准：四层检验 + 质量四问 + 铁律-1极限四问，三者全部通过
- "全面超越"已具体化为该领域可执行标准
- 质量标准可执行（替换/边界/对比三种形式至少一种）
- 穷尽性通过
- 不过拟合
- 边际成本递减
- 全文术语一致
- 门禁同步检查通过（所有执行步骤有对应门禁/检查项）
- 多文件产出物：多框架交叉审查通过（至少两种独立框架）。操作：用框架A（如模块H四步检查）审查一遍记录发现，再用框架B（如定稿清单逐项过）审查一遍记录发现，比较两次发现的差集——A发现而B没发现的、B发现而A没发现的。差集是单一框架的盲区，逐条处理。
- handbook版本对齐（引用的handbook规则是否为当前版本）

### 穷尽性检查

按五维度遍历：输入变体、输出约束、处理分支、边界条件、失败模式。每维度用正交分类。标准是"确认再查没有增量了"。

穷尽性的边界：根因是Skill规则模糊或缺失→Skill的问题；根因是模型能力上限→不是Skill的问题，不用规则补救能力缺陷。

### 过拟合三信号

禁令超总规则量25%、连续3条规则都是同一bad case的补丁、规则之间出现交叉依赖。触发后停止添加，开始合并精简。操作：从bad case提炼正面规则。

## §6 文件规范

### SKILL.md标准格式

```markdown
---
name: 英文短横线命名
description: 触发条件+功能概述（含多种说法，稍pushy）
---

# Skill名称
> 一句话核心目标

## 文风锚点（如涉及自然语言输出）
## 铁律（铁律-1标配。其余≤5条。铁律零：防编造）
## 工作流程（如非单步处理）
## 内容规范（如领域知识未充分约束产出）
## 质量标准（可执行检查方法，非口号）
```

不是所有模块都必须有，省略应是有意而非遗忘。主文件≤500行超了拆references。

### 文件结构

```
skill-name/
├── SKILL.md          # 主文件
├── references/       # 领域知识（可选）
├── scripts/          # 确定性脚本（可选）
└── assets/           # 模板等静态资源（可选）
```

### 表达与输出

分析层和输出层严格分离。核心判断变化后从零重新生成输出。

Skill设计时判断哪些产出面向读者→写入质量标准。面向读者→执行expression.md全部规则。面向AI/系统→清晰优先不执行。

**产出物形态升级（纯文字→图文排版/电子出版物）：** 通过"接口预留"而非"Skill膨胀"实现——内容Skill加视觉插槽标注规范（标注位置、类型、数据），渲染层独立处理（排版模板+图表生成+配图生成→最终产物）。P1三层分离的延伸。视觉设计决策的完整流程见`visual-design.md`——从核心洞察到决策逻辑到审美迭代到逐维度并排检验。视觉规则以决策逻辑（非固定参数）形式写入目标Skill，四层结构：决策逻辑→参考值→结构性检验→执行时审美自检。

### 迭代与重构

**重构信号：** 修改牵一发动全身、主文件超500行同时做路由和内容、审查轮次持续发散。

**经验积累接口内建：** 反模式库（表现→根因→检测→修正→实战来源）、规则标注来源和录入日期、维护记录表。

**迭代沉淀格式：** `[Skill名] 经验/教训：具体内容。建议更新：handbook的§X / 不需要。`

**记录标准：** 用户明确指出过两次以上的同类问题、导致Skill推翻重来（不是局部修改）的坑、经验证有效的审查方法或模板。用户决定是否采纳。

**经验回流：** 项目/Skill完成后复盘经验写入待验证池。每完成两个新项目review一次。

### 跨对话衔接

(1)按产出量和context预算规划对话切分 (2)每轮结束存累积状态文件（判断清单、事实清单、映射表） (3)新对话按需加载文件清单 (4)新对话第一步确认累积状态是否最新。核心：累积状态不丢失。对话中从外部获取的关键信息（搜索结果、用户提供的数据、验证结论）拿到后立即写入handoff文件，不等对话结束再整理——context会被compact，到时候关键信息可能已丢失。
